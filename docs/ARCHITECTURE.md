# 技术说明

本文档用于补充说明 Trajecta-Agent 的技术结构与工作方式。它不展开开发过程，只说明当前产品原型如何完成地点整理、约束处理与行程规划。

## 系统结构

项目由四部分组成：

- 前端：负责收集行程设置、展示地点池、展示每日路线，并承接用户后续修改
- 后端：负责解析输入、整理地点、生成路线骨架、确定性落地、校验结果并处理重排请求
- 本地存储：使用 SQLite 保存会话、地点状态、路线结果和修改记录
- 外部服务：使用 LLM 处理理解与生成，使用高德 Web 服务完成地点确认和交通信息计算

当前版本以本地运行的单体闭环为主。前端与后端分开启动，前端通过本地代理访问后端接口。

## 工作流程

Trajecta-Agent 的核心流程可以概括为：

用户输入行程信息与资料  
→ 系统整理地点  
→ LLM 生成路线骨架  
→ 系统确定性落地正式行程  
→ 校验、修复与有限重规划  
→ 生成用户可见文案  
→ 用户继续调整

展开后主要包括以下几个步骤：

1. 读取输入  
   系统接收目的地、天数、酒店名、出行人数、预算、兴趣偏好、交通偏好、路线目标和行程强度，同时接收攻略、地点清单、餐厅推荐和自由描述。

2. 解析地点与意图  
   后端从文本中提取地点、区域、餐厅、酒店和交通节点，并识别用户表达中的偏好与限制，例如优先级、节奏倾向和时间安排倾向。

3. 确认地图位置  
   系统调用高德服务，将候选地点尽量落到真实地图对象上，为后续路线规划提供可计算的位置基础。

4. 整理地点池  
   系统结合用户表达、资料证据、地点类型、酒店位置和路线目标，对地点进行初步判断，并生成可供用户少量干预的地点池。
   
   对连锁品牌，地点池阶段只表达“是否希望顺路出现”，不提前锁定具体门店。系统会保留品牌级地点和候选门店信息，把最终选店延后到规划阶段。

5. 编译规划上下文  
   后端把可规划地点、待确认地点、酒店锚点、交通矩阵、预计游玩时长、路线目标和强度预算编译成受控事实集合。`exclude` 和 `unresolved` 地点不会进入正式候选池。

6. 生成路线骨架  
   LLM 负责在候选集合内完成选点、分天和排序，只输出按天地点顺序、未安排地点、取舍原因代码和风险标签。这个阶段不生成到达时间、交通字段、总外出时长和最终文案。

7. 系统确定性落地  
   后端根据路线骨架统一生成酒店往返、点间交通、用餐段、到达时间和每日外出总时长，并同步前端需要的结构化字段。

8. 校验、修复与有限重规划  
   系统先做硬约束校验，例如超出强度、漏掉必去地点、安排了未确认地点、缺少交通边等；随后优先做规则修复。若仍失败，再把问题标签反馈给规划模型重排骨架，后台最多自动重规划 2 轮。

9. 文案生成  
   在正式事实已经稳定后，文案层再根据事实和标签生成路线概览、每日说明、未安排原因和风险提示。文案层不能新增地点、时间或风险结论。

10. 接收后续修改  
   用户可以继续提出调整要求，系统会基于当前地点池与已有路线结果进行局部重排。

如果地点池事实发生变化，例如重新识别、改名重搜或调整地点决策，已有路线结果会立即失效。这样做的目的不是强迫用户多点一次，而是避免系统在“地点事实已变化、路线事实仍停留在旧版本”时继续展示一条看似完整、实际上已经过期的路线。

## 规划层职责边界

当前版本把规划拆成两个逻辑角色：

- `planning`：负责路线骨架规划和受限重规划
- `copy`：负责把最终事实改写成用户可读文案

首版允许两个角色共用同一个 provider 和模型，但后端已经预留 `LLM_PLANNING_MODEL` 与 `LLM_COPY_MODEL` 两个可选环境变量；如果未单独配置，都会回退到 `LLM_MODEL`。

在这套边界里：

- 系统拥有地点真相、交通真相、酒店锚点、预计游玩时长、强度预算、时间线和总外出时长
- LLM 负责在候选集合内做选点、分天、排序和取舍表达
- 下游文案层不能反写上游事实

## 规划约束

Trajecta-Agent 的路线生成不是简单的地点排序，而是在一组明确约束下完成规划。当前版本主要处理以下三类约束：

### 1. 空间与时间边界

- 目的地决定整体空间范围
- 天数决定每日可分配的时间边界
- 酒店名对应每日起点和终点的参考位置

这些信息共同决定路线的基本框架。

### 2. 偏好与目标约束

- 兴趣偏好影响地点权重
- 交通偏好影响移动方式与路线成本判断
- 路线目标影响排序倾向，例如更偏向集中、顺路或特定体验优先

这一层决定“哪些地点更重要”“同样可行时优先怎样安排”。

对于连锁品牌，“顺路安排”属于路线意愿而不是门店确认。规划上下文会把它表示为品牌级候选，并附带候选门店的顺路比较信息；如果路线骨架纳入该品牌，系统再根据骨架选定具体门店，并让后续交通、时间线和用餐计算都基于真实门店。

### 3. 行程强度约束

当前版本以每日总外出时间作为强度判断口径。总外出时间覆盖：

- 酒店往返交通
- 点间交通
- 用餐时间
- 排队、游玩与休息时间

强度分为两档：

- `轻松`：7 小时以内
- `特种兵`：超过 7 小时

这项约束用于限制单日节奏，避免路线在形式上成立、实际体验上过密。用户选择轻松时，系统会尽量把每天控制在 7 小时内；如果必去地点较多或单个地点耗时较长，则优先保留必去地点，并把当天节奏显示为特种兵。

待定地点在强度较大时可以放入备选；餐饮类待定地点也必须满足顺路或低移动成本才保留。没有明确餐饮地点时，系统会预留就近用餐时间；大型场地跨饭点时，用餐可标记为场内完成，避免重复计算时长。

前端展示层对午餐和晚餐继续服从这套语义：只有当后端把某个已纳入路线的餐饮点明确承接为午餐或晚餐时，地点卡片才显示 `午餐：某餐厅` 或 `晚餐：某餐厅`；否则统一显示“就近用餐”，不把非餐饮景点误显示成用餐地点。

## 输出结果

系统最终输出的不是单一文本，而是一组可继续操作的结构化结果，主要包括：

- 当前会话的行程设置
- 整理后的地点池
- 按天拆分的路线结果
- 每日外出节奏信息
- 未安排地点与原因说明
- 风险提示与后续修改入口

这种输出方式便于用户在已有结果基础上继续调整，而不需要重新从头输入。

其中每日时间线并不是独立生成的一份“展示文案”，而是对规范化后路线 JSON 的再次可视化。前端只负责把后端已经确定的 `arrival_time`、`meal_breaks`、`transport_to_next`、酒店往返和酒店休息段绘制出来，不再额外创造隐藏缓冲或用上一站结束时间覆盖后端给出的显式到达时间。

## 当前版本边界

本文档对应的是当前本地产品原型，重点在于完成一条完整的旅行规划闭环：

- 输入用户资料
- 整理地点
- 生成路线
- 继续修改

地图展示、实时信息接入和更复杂的外部数据能力仍处于后续扩展空间中。
