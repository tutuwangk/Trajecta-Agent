# 技术说明

本文档用于补充说明 Trajecta-Agent 的技术结构与工作方式。它不展开开发过程，只说明当前产品原型如何完成地点整理、约束处理与行程规划。

## 系统结构

项目由四部分组成：

- 前端：负责收集行程设置、展示地点池、展示每日路线，并承接用户后续修改
- 后端：负责解析输入、整理地点、生成路线、校验结果并处理重排请求
- 本地存储：使用 SQLite 保存会话、地点状态、路线结果和修改记录
- 外部服务：使用 LLM 处理理解与生成，使用高德 Web 服务完成地点确认和交通信息计算

当前版本以本地运行的单体闭环为主。前端与后端分开启动，前端通过本地代理访问后端接口。

## 工作流程

Trajecta-Agent 的核心流程可以概括为：

用户输入行程信息与资料  
→ 系统整理地点  
→ 在约束条件下生成路线  
→ 校验并修正结果  
→ 用户继续调整

展开后主要包括以下几个步骤：

1. 读取输入  
   系统接收目的地、天数、酒店名、出行人数、预算、兴趣偏好、交通偏好、路线目标和行程强度，同时接收攻略、地点清单、餐厅推荐和自由描述。

2. 解析地点与意图  
   后端从文本中提取地点、区域、餐厅、酒店和交通节点，并识别用户表达中的偏好与限制，例如优先级、节奏倾向和时间安排倾向。

3. 确认地图位置  
   系统调用高德服务，将候选地点尽量落到真实地图对象上，为后续路线规划提供可计算的位置基础。

4. 整理地点池  
   系统结合用户表达、资料证据、地点类型、酒店位置和路线目标，对地点进行初步判断，并生成可供用户少量干预的地点池。

5. 生成路线  
   系统只使用满足规划条件的地点构建路线，并根据交通时间、空间关系和行程约束生成按天拆分的路线结果。

6. 校验与修正  
   初版路线生成后，系统继续检查节奏、绕路、地点冲突和约束偏离情况，再做一轮修正，最终输出路线概览、每日路线、未安排说明和风险提示。

7. 接收后续修改  
   用户可以继续提出调整要求，系统会基于当前地点池与已有路线结果进行局部重排。

## 规划约束

Trajecta-Agent 的路线生成不是简单的地点排序，而是在一组明确约束下完成规划。当前版本主要处理以下三类约束：

### 1. 空间与时间边界

- 目的地决定整体空间范围
- 天数决定每日可分配的时间边界
- 酒店名对应每日起点和终点的参考位置

这些信息共同决定路线的基本框架。

### 2. 偏好与目标约束

- 兴趣偏好影响地点权重
- 交通偏好影响移动方式与路线成本判断
- 路线目标影响排序倾向，例如更偏向集中、顺路或特定体验优先

这一层决定“哪些地点更重要”“同样可行时优先怎样安排”。

### 3. 行程强度约束

当前版本以每日总外出时间作为强度判断口径。总外出时间覆盖：

- 酒店往返交通
- 点间交通
- 用餐时间
- 排队、游玩与休息时间

强度分为三档：

- `躺平式旅游`：5 小时以内
- `常规`：9 小时以内
- `特种兵`：14 小时以内

这项约束用于限制单日节奏，避免路线在形式上成立、实际体验上过密。

## 输出结果

系统最终输出的不是单一文本，而是一组可继续操作的结构化结果，主要包括：

- 当前会话的行程设置
- 整理后的地点池
- 按天拆分的路线结果
- 每日外出节奏信息
- 未安排地点与原因说明
- 风险提示与后续修改入口

这种输出方式便于用户在已有结果基础上继续调整，而不需要重新从头输入。

## 当前版本边界

本文档对应的是当前本地产品原型，重点在于完成一条完整的旅行规划闭环：

- 输入用户资料
- 整理地点
- 生成路线
- 继续修改

地图展示、实时信息接入和更复杂的外部数据能力仍处于后续扩展空间中。
