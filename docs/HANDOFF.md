# 交接说明

## 当前状态

截至 2026-07-06，本目录已完成多阶段规划工作流重排、轻松/特种兵两档强度、后端路线规范化、餐饮与时间线真相源收口、连锁品牌顺路候选延后决策，以及地点变更后的旧路线失效处理。

已完成：

- 首页改为 `Trajecta-Agent`，保留行程强度，并新增交通偏好和路线目标。
- “旅行笔记”改为“资料与地点”，支持攻略、地点清单、餐厅推荐、酒店/交通信息和自由描述。
- 首页删除地点级“必去 / 不想去”字段，地点意图交给资料输入和地点池处理。
- 新增轻量地点池，主操作收敛为“必去、待定、移除、改名”。
- 地点池不再展示“需确认 / 已识别”地址状态；已有地图候选默认视为可用地点，错误地点通过改名重新搜索。
- 首页布局调整为左侧“行程设置”约 3/5、右侧“资料与地点”约 2/5；“识别地点”在资料模块底部，“生成路线”在识别出的地点模块底部。
- 待定地点由规划器在必去地点优先、强度上限满足后，根据空余时间机动安排。
- 后端新增 `place_organizer.py`，为每个地点生成系统默认决策、用户覆盖操作、最终决策、推断角色和前端展示对象。
- 新增 `POST /sessions/{session_id}/recognize-places` 和 `POST /sessions/{session_id}/place-overrides`，旧 `/extract-pois` 和 `PATCH /pois` 保持兼容。
- 用户不操作地点池时，也会基于系统默认决策生成路线。
- 路线结果页增加路线概览，并把未安排地点、需要确认地点和出行提醒改为折叠展示。
- 强度计算改为外出总时长口径，不按地点数量计算；当前只显示“轻松 / 特种兵”两档，以 7 小时为分界。
- 首页删除默认预填内容；`同行人` 改为“出行人数”，`酒店区域` 改为“酒店名”。
- 首页表单结构化提交 `user_profile`，酒店名保存到 `hotel_name`，后端保留文本解析兜底和旧会话回填。
- 路线目标删除“少绕路”和“轻松一点”；轻松节奏由行程强度控制。
- LLM JSON 解析层支持从代码块或说明文字中提取有效 JSON，减少“LLM 返回内容不是有效 JSON”的失败。
- 地点识别增加保守的连锁店/候选判断，“顺路安排”只通过后端白名单动作暴露到前端。
- 连锁品牌点击“顺路安排”后，地点池只记录“顺路候选”意愿，不提前替换成具体门店；具体门店在规划阶段结合整条路线和候选门店比较信息再决定。
- “识别地点”和“生成路线”按钮增加加载中反馈。
- 结果页新增代码绘制的“今日时间线”，按 5 分钟向后取整展示出发、到站、停留和点间交通预留时间。
- 时间线计算已从组件展示层拆出，优先服从后端规范化后的 `arrival_time`、`meal_breaks` 和下一站显式到达时间，不再让前端根据上一站结束时间一路漂移。
- 午餐和晚餐现已优先绑定当天已纳入路线的真实餐饮点；如果没有合适餐饮点，时间线统一显示“就近用餐”，不再把博物馆、景区等非餐饮地点显示成 `午餐：某地点`。
- Next.js 开发指示器已关闭；根布局对浏览器翻译插件注入属性启用 hydration warning 抑制。
- README 改为 `Trajecta-Agent` 产品说明，并新增 `demo/` 演示 GIF 引用。
- 前端代理支持 `BACKEND_API_BASE_URL`，跨电脑或线上部署时未配置会返回可读错误；本地开发默认代理到 `http://127.0.0.1:8000`。
- LLM 客户端支持 `LLM_TRUST_ENV` 控制是否信任系统代理环境。
- 路线修正逻辑放宽必去地点的强制删减，强度控制优先后置非必去地点；如果必去地点过多导致超出强度，会保留地点并给出风险提示。
- 后端新增统一路线规范化步骤，最终收口地点取舍、点间交通、用餐段、到达时间和每日外出总时长。
- 用户手动确认过且有地图坐标的模糊候选，在规划、交通矩阵、修正、校验和规范化阶段都按可规划地点处理。
- 待定餐饮地点只有顺路或低移动成本时才保留；远距离待定餐饮会进入备选，并由就近或场内用餐时间兜底。
- `/plan` 后端主流程已重排为“事实编译 → 路线骨架规划 → 系统落地 → 硬约束校验 → 规则修复 / 有界重规划 → 文案生成”。
- `planner.py` 已改为骨架规划器，LLM 只负责候选集合内的选点、分天、排序和取舍标签，不再直接生成完整 itinerary。
- 时间线、酒店往返、点间交通、饭点、到达时间和 `total_outing_min` 都由系统确定性生成，不再信任一次性 LLM 输出。
- 地点池事实一旦变化，例如重新识别、改名重搜或修改地点决策，旧的 `itinerary_state` 和 `revision_history` 会立即失效，前端会提示用户重新生成路线，避免把旧路线误当成最新结果。
- `verifier.py` 已拆成硬约束校验和软评审两层；软评审只产 issue tags，不直接改路线事实。
- `reviser.py` 已拆出规则修复、受限重规划和文案生成三个职责；后台最多自动重规划 2 轮。
- LLM 客户端已预留 `planning` / `copy` 两个逻辑角色，可分别通过 `LLM_PLANNING_MODEL`、`LLM_COPY_MODEL` 覆盖模型；未设置时回退到 `LLM_MODEL`。
- `/plan` 会记录 `planning_context_snapshot`、`skeleton_versions`、`hard_issue_history` 和 `repair_attempts` 调试快照到后端日志。
- 时间线规范化会在裁剪地点后同步压实 `segments`，确保前端仍能拿到保留下来站点之间的真实交通边，不会再被旧分段残留覆盖。
- `quick stop`、`meal stop` 和低绕路餐饮的保留规则已重新收口到“角色优先 + 本地绕路成本”口径，避免饮品补给和正餐候选被过强的全局裁剪规则误删。
- 固定门店但语义上属于 `light_drink`、`snack` 的地点，若已被识别为可快速停留，会在路线骨架落地时自动按 `quick stop` 处理，不再依赖连锁分支绕路元数据。

## 后续注意事项

- 如果继续修改行程强度，不要回到“地点数量”规则；轻松不等于硬删必去地点，待定地点才是高强度时优先裁剪对象。
- 选点、分天和排序仍由 LLM 主规划，但时间线、交通边、饭点和 `total_outing_min` 必须继续由系统生成，不要回退到“一次 LLM 产完整 itinerary”。
- 如果修改地点池工作流，必须继续保持“地点事实变化 = 旧路线失效”的约束，不能为了少一次点击而让前端展示过期路线。
- 如果 copy 层继续调整，必须只基于 `CopyContext` 表达已有事实，不能新增地点、时间或风险结论。
- 如果新增前端文案，先检查是否面向普通用户，避免暴露 `POI`、匹配度、后端端口和会话等词。
- 如果新增地点操作，优先判断它是否真的需要用户手动决策；默认应由后端地点组织层先判断。
- 如果修改每日时间线，只用前端代码根据路线 JSON 绘制，不让 LLM 生成可视化内容；时间展示保持 5 分钟向后取整，但不能再用前端累计逻辑覆盖后端已给出的到达时间与用餐时段。
- 如果继续调整午餐或晚餐规则，先判断当天路线里是否已有合适餐饮点可承接；只有真实餐饮点才允许显示成 `午餐：某餐厅`，否则保持“就近用餐”文案。
- 如果继续改可裁剪规则，优先保住“角色明确且局部代价低”的点，例如 `quick stop`、`meal stop`、低绕路餐饮；不要再让单一全局规则压过这些角色语义。
- 如果修改酒店名或出行人数链路，优先保持结构化 `user_profile`，不要退回“拼自然语言后再正则解析”的主路径。
- 如果增加新的 API、环境变量或数据表，需要同步 README、`AGENTS.md` 和 `docs/ARCHITECTURE.md`。
