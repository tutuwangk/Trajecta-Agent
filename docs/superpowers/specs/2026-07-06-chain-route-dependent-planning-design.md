# 连锁餐饮店顺路规划重构设计

## 目标

把当前连锁餐饮店的“顺路规划”从“系统暗中按路线猜一家门店”改成“用户先选参照点，再由系统匹配最近分店”的明确流程，并在匹配完成后立即落成具体门店，回归普通地点工作流。

本轮只解决这一条产品链路，不顺手扩展到非连锁模糊地点，也不重做整体路线规划架构。

## 设计结论

采用“确认参照点后立即落具体门店”的方案。

不采用当前仓库里已有的品牌级占位到规划阶段再选店的方案，原因有三点：

1. 这和本轮确认的新产品流程不一致。
2. 旧方案让 `arrange_nearby` 同时承担用户动作、规划语义和门店选择三层含义，容易继续引发状态错误。
3. 一旦匹配完成后立即落成具体门店，后续 POI 角色、排序、分天、交通、时间线和强度都可以直接复用现有正常链路，改动边界更清楚。

## 当前问题

当前实现的核心问题不是“顺路安排算得不准”，而是产品语义和工程状态机混在了一起：

1. 用户点击“顺路安排”后，系统会过早替用户锁定某一家门店。
2. 连锁品牌还没有落成具体门店时，就可能提前暴露“必去”“待定”等普通 POI 动作。
3. `arrange_nearby` 已经渗入地点池、数据库、规划过滤、LLM 上下文和后处理，导致它既像一次按钮点击，又像长期规划语义。
4. 当前“顺路安排”的参照逻辑更多依赖系统根据前后地点猜测，不符合本轮要求的“用户明确选一个参照点”。

## 设计原则

- 连锁店在没有指定具体分店前，不当作普通 POI 使用。
- 顺路规划必须由用户显式选择一个参照点，而不是系统暗猜。
- 一旦顺路规划完成，连锁店立即落成具体门店，并回归普通 POI 工作流。
- 前端只负责展开二次确认交互和提交参数，不自行推断状态。
- 后端是唯一真相源，地点池动作继续以后端白名单为准。
- 旧的 POI 角色、排序、时间线和归一化能力尽量复用；旧的品牌级规划占位语义停止用于这条交互链。

## 用户可见流程

### 1. 用户资料中已明确给出具体分店

- 如果用户资料里直接写出了具体分店名，系统按普通地点识别。
- 该地点直接进入正常识别和规划流程，不进入顺路规划分支。

### 2. 只识别出品牌名，尚未指定具体分店

- 连锁品牌识别成功后，地点卡片只显示：
  - `顺路规划`
  - `改名`
  - `移除`
- 此时不显示：
  - `必去`
  - `待定`
- 可显示轻提示，例如“还没有确定具体门店”。

### 3. 用户点击顺路规划

- 地点卡片内展开一个二次确认区域，不跳页：
  - 一个下拉框，列出允许作为参照点的地点
  - 一个确认按钮
  - 一个取消按钮
- 允许作为参照点的地点包括：
  - 已识别且非连锁的正式地点
  - 酒店
  - 待定地点

### 4. 用户确认参照点

- 后端基于该参照点的坐标，在该连锁店已有 `candidate_options` 中匹配最近分店。
- 系统返回用户可见结果：
  - `最顺路的店是：（具体门店名）`
- 该连锁店立即从品牌级条目变成具体门店。

### 5. 顺路规划完成后

- 这家店作为普通已识别地点显示：
  - `顺路规划`
  - `必去`
  - `待定`
  - `改名`
  - `移除`
- 可加一条轻提示，例如“按某某地点顺路匹配”。
- 从这一步开始，它像其他普通 POI 一样进入路线规划。

### 6. 重新顺路规划

- 已经匹配成具体门店后，仍允许再次点击 `顺路规划`。
- 用户可以重新选择参照点。
- 新匹配结果覆盖旧的参照点绑定和具体门店结果。

## 状态机

### 状态 1：`chain_unresolved`

含义：

- 连锁品牌已识别，但还没有指定具体分店。

用户动作：

- `顺路规划`
- `改名`
- `移除`

说明：

- 该状态不显示 `必去` 或 `待定`。
- 不进入普通 POI 的排序、分天和用餐落位语义。

### 状态 2：`chain_resolved_from_anchor`

含义：

- 用户已指定参照点，系统已匹配出一个具体分店。

说明：

- 对数据库来说，这条记录仍然知道自己来自顺路规划。
- 对路线规划来说，它已经是普通 POI。

### 状态 3：`normal_grounded_poi`

含义：

- 对规划器、归一化、校验器和前端路线展示来说，该条目按普通已识别地点处理。

说明：

- 这是运行时语义，不要求数据库再复制一套独立记录类型。
- 重点在于：只要 `chain_status="resolved"`，后续主链路就不需要再理解“品牌级占位店”。

## 后端数据设计

保留当前 `grounded_poi` 为主要载体，并新增或稳定以下字段：

- `is_chain: boolean`
- `chain_status: "unresolved" | "resolved"`
- `candidate_options?: list`
- `resolved_branch_id?: string`
- `resolved_branch_name?: string`
- `resolved_from_anchor_poi_id?: string`
- `resolved_from_anchor_name?: string`
- `resolved_by?: "nearby_anchor"`

字段语义：

- `is_chain=true` 且 `chain_status="unresolved"`：
  还是品牌级条目，未确认具体门店。
- `is_chain=true` 且 `chain_status="resolved"`：
  已落成具体门店，但保留“由顺路规划得到”的来源信息。
- `candidate_options`：
  仍来自识别阶段的候选分店列表，作为顺路匹配的搜索池。
- `resolved_from_anchor_*`：
  用于后续回退和重绑判断。

## API 与动作流程

### 总体原则

不再让 `decision=arrange_nearby` 直接代表“长期规划语义”。

这次把“顺路规划”改成一个明确的两步动作：

1. 前端展开参照点选择
2. 后端确认参照点并立即落成具体门店

### 方案选择

为了减少接口膨胀，优先推荐以下落地方式：

- 前端点击 `顺路规划` 时，只在本地展开下拉框，不单独请求后端。
- 前端点击确认时，请求一个新的明确动作，例如：
  - `decision="confirm_arrange_nearby"`
  - 同时提交 `anchor_poi_id`

如果现有 `PoiDecisionUpdate` 结构不适合承载 `anchor_poi_id`，则为这条链路新增一个专用接口也可以，但要保持语义清晰。

### 确认顺路规划时的后端处理

后端收到 `poi_id + anchor_poi_id` 后，执行以下步骤：

1. 校验当前地点必须是连锁店。
2. 校验其当前状态允许顺路规划：
   - `chain_status="unresolved"`，或
   - `chain_status="resolved"` 但用户在重新绑定。
3. 校验 `anchor_poi_id` 来自允许列表：
   - 已识别非连锁正式地点
   - 酒店
   - 待定地点
4. 从该连锁店的 `candidate_options` 中，按参照点坐标匹配最近门店。
5. 返回用户可见结果：
   - `matched_branch_name`
   - `matched_branch_id`
   - `anchor_poi_name`
   - `message = "最顺路的店是：..."`

### 顺路规划成功后的落库结果

顺路规划成功后，直接把该连锁店改写成“具体门店版 grounded_poi”：

- `standard_name` 改成具体店名
- `amap_id` 改成具体分店 id
- `address`、`location` 改成具体分店信息
- `chain_status="resolved"`
- 写入：
  - `resolved_branch_id`
  - `resolved_branch_name`
  - `resolved_from_anchor_poi_id`
  - `resolved_from_anchor_name`
  - `resolved_by="nearby_anchor"`

## 决策同步规则

本轮只保留**单向联动**：

- 参照点变更可以影响连锁店
- 连锁店自身变更不反向影响参照点

### 顺路规划成功当下的同步

在 `confirm_arrange_nearby` 成功的那一刻：

- 如果参照点当前是 `must_include`，连锁店自动设成 `must_include`
- 如果参照点当前是 `optional`，连锁店自动设成 `optional`
- 如果参照点当前是普通 `include`，连锁店保持普通 `include`

### 连锁店自身后续改动

匹配完成后，如果用户直接把这家连锁店改成 `必去` 或 `待定`：

- 只改这家店本身
- 不回写参照点

## 回退与重绑规则

### 参照点被移除

如果某个参照点被改成 `remove`：

- 找出所有 `resolved_from_anchor_poi_id == 该参照点` 的连锁店
- 这些连锁店自动回退到 `chain_unresolved`
- 清空：
  - `resolved_branch_id`
  - `resolved_branch_name`
  - `resolved_from_anchor_poi_id`
  - `resolved_from_anchor_name`
  - 具体分店坐标与门店信息
- `standard_name` 恢复为品牌名或“待选择”态
- 清掉这家店因顺路规划带来的当前决策状态，回到初始普通状态
- 同时使旧的 `itinerary` 和 `revision_history` 失效

### 参照点改成必去

如果某个参照点被改成 `must_include`：

- 已绑定连锁店不回退
- 直接同步成 `must_include`

### 重新顺路规划

如果用户对已落店的连锁店再次点击 `顺路规划`：

- 允许重新选择参照点
- 新门店覆盖旧门店
- 新的 `resolved_from_anchor_*` 覆盖旧绑定

## 前端交互设计

### unresolved 连锁店卡片

只显示：

- `顺路规划`
- `改名`
- `移除`

不显示：

- `必去`
- `待定`

### 点击顺路规划后的展开区域

卡片内展开：

- 单选下拉框
- 确认按钮：`确认顺路规划`
- 取消按钮

这一步是一次二次确认，不应被包装成普通 POI 决策按钮。

### resolved 后的卡片

显示：

- `顺路规划`
- `必去`
- `待定`
- `改名`
- `移除`

并且可以显示一条轻提示，例如：

- `按“武侯祠”顺路匹配`

### 前端边界

- 前端不自行推断哪些状态该联动。
- 前端继续以 `primary_actions` 为按钮白名单真相源。
- 前端只负责提交 `anchor_poi_id` 和刷新后端返回结果。

## 旧逻辑取舍

### 保留

以下旧能力继续保留：

1. POI 角色定位
   - 连锁店落成具体门店后，餐饮角色、用餐时段匹配等能力继续复用。
2. 排序与分天框架
   - 一旦落成普通 POI，继续进入现有 planner 主链路。
3. 路线后处理
   - 交通边、到达时间、用餐、强度和 itinerary normalization 保持现有思路。
4. 地点池动作白名单
   - 前端继续按后端返回的 `primary_actions` 渲染按钮。

### 停止用于这条交互链

以下旧逻辑不再作为本条链路的核心设计：

1. 不再用 `route_dependent_chain` 把品牌级条目带进规划阶段再选店。
2. 不再让 `arrange_nearby` 承担长期规划语义。
3. 不再让未落店的品牌级连锁店参与 `必去/待定` 和正常排序。
4. 不再根据“前后已确认地点”自动暗选门店，新的唯一参照源是用户明确选择的 `anchor_poi_id`。

## 测试与验收标准

### 后端回归测试

至少覆盖以下场景：

1. 未指定分店的连锁店
   - `primary_actions` 只有 `顺路规划 / 改名 / 移除`
   - 没有 `必去 / 待定`

2. 顺路规划成功
   - 返回“最顺路的店是：xxx”
   - `standard_name / amap_id / address / location` 更新为具体门店
   - `chain_status` 从 `unresolved` 变 `resolved`
   - `resolved_from_anchor_poi_id` 正确写入
   - 若参照点是 `optional`，该店自动变 `optional`
   - 若参照点是 `must_include`，该店自动变 `must_include`

3. 参照点移除触发回退
   - 连锁店恢复为品牌级条目
   - 清空具体门店和 anchor 绑定字段
   - `itinerary` 与 `revision_history` 失效

4. 参照点改成必去
   - 已绑定连锁店不回退
   - 自动同步成 `must_include`

5. 连锁店自身改动不反向影响参照点
   - 连锁店改成 `必去/待定` 不回写参照点

6. 重新顺路规划
   - 已 resolved 的连锁店允许换 anchor
   - 新门店结果覆盖旧绑定

### 前端交互测试

至少覆盖以下场景：

1. unresolved 连锁店不显示 `必去 / 待定`
2. 点击 `顺路规划` 后展开下拉和确认按钮
3. 确认成功后，卡片切成普通 POI 视图
4. resolved 后仍可再次顺路规划
5. 参照点移除后，页面刷新能看到它回到未确定具体门店状态

### 最终验收标准

1. 用户资料里直接写明具体分店时，系统按普通 POI 识别，不进入顺路规划。
2. 用户只写品牌名时，识别后不会提前落成具体分店。
3. 未落店前，连锁店不能被标成必去或待定。
4. 用户完成顺路规划后，系统明确返回最顺路的具体门店名。
5. 顺路规划完成后，这家店作为普通 POI 进入排序、分天、交通、时间线和强度计算。
6. 参照点被移除时，绑定连锁店自动回退。
7. 参照点改成必去时，绑定连锁店自动同步为必去，但不回退。
8. 连锁店自身的必去/待定调整不会反向影响参照点。

## 非目标

- 不扩展到非连锁模糊地点的自动门店选择。
- 不在这一轮引入复杂 NLP 的顺序关系理解。
- 不重做首页整体布局。
- 不顺手重构路线规划主链路中与本需求无关的部分。
