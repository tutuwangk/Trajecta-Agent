# 2026-07-07 路线规划协议脆弱点排查

## 背景

前端正常使用流程里，生成路线返回了 `LLM 返回了不支持的 meal slot source`。这说明问题不在用户操作，而在**后端把 LLM 输出协议校验得过于脆弱**：模型语义基本正确，但字段枚举值稍微偏一点，就会直接中断整条规划链路。

本次排查目标不是放松所有校验，而是区分两类边界：

- **必须继续硬拦截**：候选集合之外的地点、重复排点、缺少关键 `poi_id`、非法连锁门店等事实越界。
- **应该归一化或降级处理**：中英文/同义词/破折号/空格变体、可选提示字段、逗号分隔字符串数组。

## 排查范围

重点检查了后端所有会直接消费 LLM 输出并可能中断主流程的入口，结论如下：

- `backend/app/agents/planner.py`
  - 风险最高。
  - 这里负责解析路线骨架、蓝图重排、`meal_slots`、`segments`、`scheduled_roles`、`selected_branch_ids`。
  - 也是本次真实报错的来源。
- `backend/app/agents/verifier.py`
  - 不消费严格的 LLM 结构化协议；主要做规则校验。
  - 软评审 LLM 失败时会静默回退到已有规则，不会产生同类硬报错。
- `backend/app/agents/reviser.py`
  - 文案/改路线阶段有回退逻辑，不是这类主因。
- `backend/app/agents/visit_duration_estimator.py`
  - 时长估计允许回退到规则值，不是高风险协议边界。
- `backend/app/agents/ugc_reader.py`
  - 解析较松，不是当前“正常流程直接中断”的关键来源。

结论：**同类风险主要集中在 `planner.py` 的 LLM 结构化结果归一化边界。**

## 查出的具体问题与根因

### 1. `meal_slots.source` 只接受精确枚举

- 现象
  - 模型只要输出 `restaurant`、`nearby`、`onsite`、`inside venue` 这类语义正确但不完全等于代码枚举的值，就会直接抛错。
- 根因
  - 提示词用的是“真实餐饮地点 / 场内用餐 / 就近补位”这样的语义描述。
  - 解析器只接受 `poi`、`inside_poi`、`fallback_nearby` 三个精确字符串。
  - **提示语义和代码协议没有被同一层严格绑定。**
- 处理
  - 增加常见同义值归一化。
  - 当 `source` 不可靠但结构字段存在时，优先按 `within_poi_id` / `poi_id` 推断来源。
  - 当完全无法判断时，降级为 `fallback_nearby`，避免直接中断。

### 2. `segments.kind` 只接受 `outing` / `hotel_rest`

- 现象
  - 模型若输出 `visit`、`rest`、`hotel-rest` 之类非常自然的变体，会直接抛 `不支持的 segment kind`。
- 根因
  - 解析器使用了精确字符串匹配。
  - 提示词虽然给了 schema，但没有充分抵消模型改写枚举名的倾向。
- 处理
  - 对常见别名做归一化。
  - 当 `kind` 不可靠时，按字段形状推断：
    - 有 `poi_ids` 就视为 `outing`
    - 有 `duration_min` / `reason` 就视为 `hotel_rest`

### 3. `meal_slots.slot` / `requirement` 对语言和别名过敏

- 现象
  - `slot` 输出成 `午餐` / `晚餐`，或 `requirement` 输出成 `must` / `needed`，此前都会直接失败。
- 根因
  - 提示语境以中文为主，但 schema 又要求英文枚举。
  - 解析器没有做中文到英文、语义别名到标准枚举的桥接。
- 处理
  - `slot` 支持中英文标准映射。
  - `requirement` 对未知值默认降级为 `required`，不再因为可选语义词差异中断主流程。

### 4. `scheduled_roles` 把“可选提示”当成“必需正确字段”

- 现象
  - 模型返回 `anchor`、`meal`、`nightlife` 这类明显可映射的角色，会被判为非法。
  - 更糟的是，即便角色完全无法识别，也会直接中断路线，而不是走系统已有的默认推断。
- 根因
  - `scheduled_roles` 本质上只是一个**增强提示**。
  - 但旧实现把它当成了必须完全符合协议的结构化字段。
  - 其实后续 `_resolved_scheduled_role` 已经能基于餐饮 slot、地点类型和语义自动推断默认角色。
- 处理
  - 对常见别名归一化，例如 `anchor -> anchor_visit`、`meal -> meal_stop`。
  - 对无法识别或未安排地点上的角色，直接忽略，不再抛错。

### 5. `selected_branch_ids` 校验过严，而且校验源本身有偏差

- 现象
  - 只要模型返回了任何不合法或多余的 `selected_branch_ids`，旧逻辑就可能直接失败。
- 根因
  - 这个字段本来也是可选增强提示。
  - 旧实现对未安排地点和无效分店 id 直接抛错。
  - 同时，旧代码构造 `branch_options_by_poi` 时读的是 `plannable_pois`，而这里通常并不保留 `route_branch_options`，导致有效分店集合来源不稳。
- 处理
  - 改为从 `poi_lookup` 读取真实门店候选。
  - 对未安排地点或无效门店 id 的返回值直接忽略。
  - 这样即便模型乱填，也会回退到系统默认门店选择，不会把整条路线打断。

### 6. 数组字段对“逗号分隔字符串”不够宽容

- 现象
  - 模型把 `poi_ids`、`risk_tags`、`reason_codes` 这类数组写成 `"a, b, c"` 时，旧逻辑会把它当成单个字符串元素处理。
  - 在 `poi_ids` 等字段上，这会进一步触发“候选集合之外的 poi_id”之类的假阳性错误。
- 根因
  - `_string_list` 默认假设模型一定返回 JSON 数组。
  - 实际上，模型在结构化输出里偶尔会把短数组压成单个逗号分隔字符串。
- 处理
  - `_string_list` 现在支持按换行、英文逗号、中文逗号、顿号拆分。

## 本次代码层处理

本次只对 `backend/app/agents/planner.py` 做了协议边界加固，原则是**最小改动**：

- 在提示词里补充了关键枚举的显式允许值。
- 对 `segments.kind`、`segment_time`、`meal_slots.slot`、`meal_slots.requirement`、`meal_slots.source`、`scheduled_roles` 做归一化。
- 对 `selected_branch_ids` 和 `scheduled_roles` 这类可选提示字段，改为“无效就忽略”，不再整条链路失败。
- 修正 `selected_branch_ids` 的可用门店来源，改从 `poi_lookup` 读取。
- 对数组字符串做分隔兼容。

## 仍然保留的硬拦截

以下校验**没有被放宽**，因为它们属于事实越界，不应该被默默吞掉：

- payload 不是对象
- day 节点不是对象
- `poi_id` 不在候选集合
- 同一天重复安排同一 `poi_id`
- 多天重复安排同一 `poi_id`
- `meal_slots.source=poi` 但缺少 `poi_id`
- `meal_slots.source=inside_poi` 但缺少 `within_poi_id`
- `meal_slots` 绑定的 `poi_id` / `within_poi_id` 不在允许集合
- `unscheduled` 里的 `poi_id` 不在候选集合

这些错误如果发生，说明不是“协议词稍微偏了”，而是**路线事实已经不可信**。

## 结论

这次真实暴露出来的，不是某一个孤立报错，而是路线规划层的**LLM 协议边界过于脆弱**。根因不是业务规则太复杂，而是：

- 提示词用语义描述
- 解析器用精确枚举
- 可选提示字段被当成硬结构字段
- 模型常见的自然变体没有被归一化

本次处理后，主流程对常见协议漂移已经更稳，但依然保留了对事实越界的硬保护。这一层现在更接近“能跑通真实用户流程”的状态。
