# 产品设计基本思路

## 1. 产品定位

Trajecta-Agent 是面向真实旅行决策场景的路线整理 Agent。用户输入目的地、天数、住宿区域、同行人、预算、兴趣偏好，以及攻略、地点清单、餐厅推荐或旅行想法后，系统自动识别地点，匹配真实位置，并生成可执行路线。

产品的一句话定义：

```text
把用户想去的地方整理成一份能直接出发的旅行方案。
```

产品设计遵循一个基本原则：用户只需要表达“我要去哪、有什么资料、偏好是什么”，系统负责完成地点识别、位置确认、路线取舍、时间安排和风险提示。用户只在关键不确定处做少量判断。

## 2. 第一性原理

旅行规划产品的核心价值是降低用户从“想去很多地方”到“知道每天怎么走”的决策成本。

用户需要的是：

```text
哪些地方值得放进去
这些地方怎么排才顺路
每天会不会太赶
哪些地方需要删掉或后置
遇到下雨、排队、改酒店时怎么调整
```

用户不需要理解系统内部的 POI 匹配、来源类型、置信度、距离矩阵、规划器、校验器和修正器。前端只展示会影响用户决策的信息，后端保留完整结构化过程。

产品交互应控制在三步：

```text
输入资料
→ 系统识别并整理
→ 生成路线并继续修改
```

地点确认环节采用轻量模式。系统默认处理高置信度地点，只把需要用户判断的地点、用户可能强烈关心的地点、可能影响路线取舍的地点展示出来。

## 3. 输入体系

首页只收集生成路线所需的最低信息。

### 3.1 基础行程信息

保留字段：

```text
目的地
天数
酒店区域
同行人
预算
兴趣偏好
交通偏好
路线目标
```

这些字段用于确定路线边界、候选地点权重和交通方式偏好。

其中：

```text
目的地、天数、酒店区域：决定空间和时间边界
同行人、预算、兴趣偏好：决定地点权重和体验排序
交通偏好：决定路线成本计算方式
路线目标：决定规划器的优化方向
```

首页不收集必去、不去等细粒度地点决策。系统识别候选地点后，再在地点层面提供轻量选择。

### 3.2 多源资料输入

右侧输入区统一命名为“资料与地点”。输入内容可以包括：

```text
攻略内容
地点清单
餐厅推荐
酒店地址
交通节点
朋友建议
用户自然语言旅行想法
```

系统将这些内容统一解析为 Source，不要求用户提前说明来源类型。

### 3.3 用户意图抽取

用户在输入框中自然表达的意图由后端抽取。例如：

```text
“太古里必须去” → 太古里优先级提升
“建设路可去可不去” → 建设路默认可选
“别去太远” → 降低跨区移动
“晚上想看夜景” → 夜景地点权重提升
“酒店在春熙路” → 作为每日起终点参考
```

这些意图不需要在首页做成独立字段。

## 4. Agent 总体框架

系统采用“多源输入解析 + 自动地点整理 + 约束感知规划 + 校验修正”的 Agent 框架。

```text
User Input
基础行程信息 + 资料与地点
        ↓
Input Parser
解析目的地、天数、酒店区域、同行人、预算、交通偏好、路线目标和兴趣偏好
        ↓
Source Reader
读取用户输入的攻略、地点清单、餐厅推荐和自然语言需求
        ↓
Place & Intent Extractor
抽取候选地点、区域、餐厅、酒店、交通节点、偏好和限制
        ↓
POI Grounder
调用地图服务匹配真实地点，生成标准 POI
        ↓
Auto Place Organizer
按置信度、用户表达、来源证据和地点类型自动整理地点池
        ↓
Light Decision Layer
只暴露必要选择：保留、一定去、移除、改名确认
        ↓
Route Matrix Builder
计算地点之间的交通时间、距离和空间关系
        ↓
Constraint-aware Planner
结合路线目标、交通偏好、兴趣偏好、住宿区域和地点权重生成路线
        ↓
Verifier
检查路线可执行性、地点冲突、交通成本、餐饮安排和未匹配地点
        ↓
Reviser
根据校验结果修正路线，生成取舍说明
        ↓
Itinerary Formatter
输出路线概览、每日路线、未安排原因、风险提示和修改入口
        ↓
Revision Loop
根据用户反馈进行局部重排
```

## 5. 关键模块设计

### 5.1 Input Parser

负责解析首页基础字段和输入框中的自然语言约束。

输出 PlanningConstraints：

```json
{
  "destination": "成都",
  "days": 2,
  "hotel_area": "春熙路",
  "travelers": "情侣",
  "budget_level": "medium",
  "transport_preference": ["walk", "taxi"],
  "route_goal": "food_first",
  "interest_preferences": ["美食", "拍照"]
}
```

该模块需要同时读取结构化表单和输入框文本，补全用户自然表达的隐藏约束。

### 5.2 Source Reader

负责将“资料与地点”输入框内容解析为 SourceItem。

```json
{
  "source_id": "source_001",
  "source_type": "free_text",
  "raw_text": "用户输入的原始内容",
  "summary": "内容摘要",
  "source_confidence": 0.85
}
```

当前版本不需要在前端展示 SourceItem。SourceItem 主要用于后端保留证据和支持后续解释。

### 5.3 Place & Intent Extractor

负责从 SourceItem 中抽取地点和意图。

输出 CandidatePlace：

```json
{
  "candidate_id": "cand_001",
  "raw_name": "IFS",
  "source_contexts": ["想去 IFS 拍照"],
  "place_type": "shopping",
  "experience_tags": ["拍照", "商圈"],
  "extracted_intent": "strong_interest",
  "initial_confidence": 0.86
}
```

该模块需要区分真实地点、区域、餐厅、酒店、交通节点、商品名、体验词和泛称。

### 5.4 POI Grounder

负责调用地图服务完成地点标准化。

输出 GroundedPlace：

```json
{
  "candidate_id": "cand_001",
  "standard_name": "成都IFS国际金融中心",
  "amap_id": "B001C7XXXXX",
  "address": "成都市锦江区红星路三段1号",
  "location": {
    "lng": 104.080000,
    "lat": 30.657000
  },
  "district": "锦江区",
  "match_confidence": 0.92,
  "match_status": "matched"
}
```

匹配状态：

```text
matched：可直接用于规划
ambiguous：需要轻量确认
unmatched：进入未确认清单，不进入正式路线
```

### 5.5 Auto Place Organizer

负责把候选地点整理成前端可理解的轻量地点池。

系统自动生成 `system_decision`：

```text
include：默认纳入规划
optional：作为备选或低优先级地点
needs_confirmation：需要用户确认
exclude：默认不进入规划
```

判断依据：

```text
用户明确表达
地图匹配置信度
来源上下文强度
地点类型
与目的地和酒店区域的关系
与路线目标的匹配程度
```

前端默认只展示地点名称、类型、状态和简短操作。匹配细节、来源证据和置信度进入展开信息或调试信息，不作为主要界面内容。

### 5.6 Light Decision Layer

用户可进行的核心操作控制在三类：

```text
一定去
移除
改名确认
```

系统默认保留高置信度地点。用户无需逐一确认全部地点。

地点角色由后端自动推断：

```text
餐厅 → meal
酒店/住宿区域 → hotel
机场/车站 → transport
景点/商圈/街区 → visit
用户明确起点/终点 → start/end
```

当前版本前端可暂不展示复杂角色选择。

### 5.7 Route Matrix Builder

负责计算地点之间的交通时间和距离，形成路线规划所需的空间关系。

关系类型：

```text
same_cluster：适合同半天连续安排
nearby：适合同一天安排
same_day_possible：可同一天安排，但要控制移动成本
separate_day：建议拆分或作为备选
unknown：路径信息不足
```

路线顺路性以地图路径结果为准。

### 5.8 Constraint-aware Planner

负责生成初版路线。

规划器需要处理：

```text
路线目标
交通偏好
兴趣偏好
住宿区域
地点权重
地图距离
餐饮时段
每日可用时间
地点可执行性
```

地点过多时，系统主动完成筛选。筛选结果需要进入 `unscheduled_places`，用于解释哪些地点未安排以及原因。

### 5.9 Verifier

负责检查路线是否可执行。

检查项：

```text
unmatched 地点是否进入路线
ambiguous 地点是否未经处理进入路线
地点顺序是否绕路
交通时间是否过长
餐饮时间是否合理
路线是否遗漏高优先级地点
是否安排了用户移除的地点
未安排地点是否有原因
```

Verifier 输出结构化问题清单，供 Reviser 修正。

### 5.10 Reviser

负责修正路线并生成解释。

修正动作：

```text
删除低权重地点
调整访问顺序
移动地点到另一天
降低交通成本
补充餐饮安排
替换不确定地点
生成备选方案
整理未安排原因
```

### 5.11 Revision Loop

用户在路线结果页只需要通过自然语言或少量快捷按钮提出修改。

建议保留：

```text
轻松一点
多安排美食
增加夜景
删掉某个地点
换酒店区域
下雨方案
```

系统基于当前地点池、路线结果和用户反馈进行局部重排。

## 7. 前端信息展示原则

前端只展示用户做决策所需的信息。

### 7.1 地点池展示原则

地点池不做表格化重展示。建议采用简洁卡片或标签列表：

```text
地点名
类型标签
状态标签
主要操作
```

状态标签建议：

```text
已识别
需确认
未匹配
```

主要操作：

```text
一定去
移除
改名
```

地点池默认折叠低价值细节。来源、地址、匹配候选等信息放入展开态。

### 7.2 路线结果展示原则

路线结果首页优先展示：

```text
路线主题
每天怎么走
预计交通时间
关键提醒
继续修改入口
```

未安排地点、风险提示、不确定地点采用折叠区块，避免干扰主路线阅读。

### 7.3 修改入口原则

快捷修改按钮保留少量高频动作。复杂修改交给自然语言输入框。

## 8. 后端设计原则

后端承担更多自动判断，降低前端交互负担。

### 8.1 默认决策

后端需要为每个地点生成系统默认决策：

```text
高置信度、强用户意图、与路线目标匹配 → include
高置信度、弱相关、时间可能不足 → optional
匹配不确定、同名地点较多 → needs_confirmation
无可靠地图结果或用户表达规避 → exclude
```

### 8.2 自动角色推断

后端需要推断地点角色：

```text
餐饮类 → meal
酒店/住宿区域 → hotel
机场/车站 → transport
景点/商圈/街区 → visit
用户文本中出现“从这里出发” → start
用户文本中出现“最后到这里” → end
```

### 8.3 规划可直接运行

用户点击“识别地点”后，系统应返回可直接生成路线的地点池。用户没有额外操作时，点击“生成路线”也能产出合理结果。

### 8.4 只在必要时打断用户

以下情况需要用户确认：

```text
关键地点匹配不唯一
用户输入的地点无法匹配
同一个名称在目的地有多个常见候选
系统推断的酒店/起点/终点不稳定
必然影响路线取舍的地点存在歧义
```

其余情况由系统先做默认判断，并在路线结果中解释。

## 9. 输出原则

最终输出应形成可执行方案，而非展示系统过程。

路线结果至少包含：

```text
路线概览
每日路线
每段交通时间
每个地点停留时间
关键提醒
未安排地点及原因
一键修改入口
```

未安排原因使用简洁表达：

```text
距离较远
时间不足
与本次路线目标关联较弱
匹配不确定
更适合作为单独半天
与已安排地点类型重复
```

## 10. 当前版本目标

当前版本的目标是实现一个低操作成本的旅行路线整理 Agent：

```text
用户输入基础行程信息和旅行资料后，系统自动识别地点、完成地图匹配、整理候选地点、生成路线，并允许用户用少量操作继续调整。
```

本版本优先保证：

```text
首页输入简洁
地点池轻量确认
系统默认可规划
路线结果清楚
修改入口直接
后端自动判断充分
```
