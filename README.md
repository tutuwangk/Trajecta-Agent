# Trajecta-Agent

把零散旅行资料和想法整理好，给你一份每日行程规划。

Trajecta-Agent 面向真实旅行决策场景。用户输入目的地、天数、交通偏好、路线目标和行程强度等信息，并粘贴攻略、地点清单、餐厅推荐或自由描述后，系统会整理地点、确认位置，并生成可继续调整的每日路线。


## 功能简介

- 收集目的地、天数、酒店名、出行人数、预算、兴趣偏好、交通偏好、路线目标和行程强度
- 识别资料里的景点、餐厅、酒店和其他地点
- 生成轻量地点池，让用户用少量操作完成关键决策
- 生成按天拆分的旅行路线
- 在结果页继续修改，让 Agent 重新整理
- 连锁品牌可先做“顺路规划”，先选参考地点，再匹配最近的具体门店

适合的输入包括攻略摘录、餐厅清单、朋友推荐、酒店地址、备忘录，以及自然语言形式的旅行想法。

## Agent 工作流程


1. 收集行程约束  
   前端同时提交结构化行程设置与原始资料。结构化部分包括目的地、天数、酒店名、出行人数、预算、兴趣偏好、交通偏好、路线目标和行程强度；原始资料部分承载攻略、地点线索、餐厅推荐和补充说明。两类输入共同构成规划约束。

2. 解析资料与用户意图  
   后端从原始资料中抽取候选地点、区域、餐厅、酒店、交通节点以及用户意图。这一步采用“宽进严出”：识别模型负责高召回，短品牌、中英混合名称、带符号名称和未指定分店的连锁品牌都会先保留；原文中的“去、吃、喝、逛、看、买、顺便、再”等明确意图还会做确定性回收。识别模型暂时失败时，原文回收仍可继续，候选最终由地图确认和地点池过滤，不会因为提高召回而直接污染正式路线。

3. 完成地图匹配与地点整理  
   系统调用高德服务，将候选地点尽量落到真实地图对象上，并结合来源证据、用户表达、地点类型、酒店位置、路线目标和兴趣偏好，为每个地点生成默认决策。整理后的地点池保留“必去、待定、移除、改名”等轻量操作，用户可在关键处补充判断。

   如果某个地点是连锁品牌，地点池会先显示“顺路规划 / 改名 / 移除”。点击“顺路规划”后，用户先从已识别地点、酒店或待定地点里选一个参考地点，系统再立即匹配最近的具体门店。匹配完成后，这家店会作为普通地点进入后续路线规划。

4. 生成路线蓝图
   进入正式规划的地点，需要同时满足地图确认和最终决策要求。规划阶段会综合考虑以下因素：
   - 天数与每日时间边界
   - 酒店作为每日起终点的参考位置
   - 交通偏好对应的移动成本
   - 路线目标对应的排序倾向
   - 兴趣偏好与地点权重
   - 行程强度与每日外出时间

   系统先根据地点类型和资料证据整理正餐能力与适合时段，再用一次批量、non-thinking 的轻量 LLM 调用估计所有已确认地点的正常游玩时长。时长模型先判断完整正餐、目的型购物、饮品补给或大型景点等实际活动角色，再估计停留；系统用角色级合理区间拦住明显偏短或偏长的单点结果，并柔性忽略不兼容的 `quick_stop` 等角色，不把模型漂移升级成规划报错。模型、缓存或返回协议失败时整批或单点回退到本地时长规则，不阻断路线生成。LLM 再在这个稳定事实快照上生成按天拆分的路线蓝图，负责选点、分天、分段、饭点策略、排序和取舍。蓝图必须完整返回用户要求的全部天数，但不直接生成交通时间、完整时间线和最终文案。

   发送给规划模型的内容会先编译成唯一的 `PlanningPacket`：每个地点只保留名称、区域、优先级、规划角色、游玩时长、餐饮能力、适合时段和有限近邻关系。完整地图候选、重复语义字段与两两路线矩阵留在系统内部，不重复塞进提示词。

5. 系统落地、校验并修正
   后端只为蓝图实际采用的相邻地点请求高德精确路线，再把蓝图确定性编译为酒店往返、点间交通、饭点、到达时间和每日外出总时长。初稿和最多两次重排都会保留，系统发布事实合法且综合质量最高的一版；普通时段、顺序、空档和饭点体验偏差只影响评分与提示，不再堵死流程。LLM 输出为空、截断或不符合协议时会先重试，仍失败则使用真实地点与地图事实生成稳定蓝图。

   前端时间线直接消费后端交通事实，连续的上午、下午、傍晚或夜间分段之间也会保留相邻交通；真正回酒店休整时才切换为酒店往返。模型文案进入页面前会在后端和前端两层清理显式 thinking / analysis / reasoning 内容及内部字段，用户只看到最终路线说明。

   规划模型会显式使用 thinking 模式；时长估计使用 non-thinking 和 `temperature=0.1`，文案、软评审和资料解析默认使用 non-thinking。JSON 截断时会扩大一次输出预算后重试。每次调用只记录模型、模式、token、耗时、缓存命中、结束原因和错误类型，不保存 prompt 正文。

   当单个地点停留至少 4 小时并实际覆盖午餐或晚餐时间，且当天没有合适的真实餐厅时，系统会把泛化的就近用餐确定性落到该地点场内或紧邻位置，并把用餐包含在地点停留时长内，避免为了外部用餐延迟大型景区入场。

   如果必去地点、明确时段、正餐安排或强度之间出现明显冲突，系统按“明确要求 > 必去 > 正餐 > 普通地点 > 待定 > 舒适度”的优先级自动收敛；强度偏高和跨区偏重会作为提示，不再要求用户处理系统可以自动完成的取舍。

6. 根据反馈继续重排  
   用户可以通过快捷操作或自然语言继续提出修改，例如调整某一天的节奏、更换酒店、后置某个地点或重新做连锁店的顺路规划。系统会基于当前地点池、约束条件和既有路线结果进行局部重排。

   如果用户重新识别地点、改名重搜，或修改地点池决策，系统会把旧路线视为过期结果，需要重新点击“生成路线”。这条提示不会禁用生成按钮。正常生成流程不再返回“需要你取舍”卡片。

每次规划都会生成 `run_id`，并记录当前阶段、尝试次数、耗时和结构化错误摘要，便于区分模型协议、地图事实、时间编译和最终校验问题。

如果同步规划请求在长时间运行时被前端代理中断，页面会继续轮询本次 `planning_run`；后端完成后自动读取已保存路线，不重复提交规划请求。

## 本地部署

需要本地安装：

- Python 3
- Node.js
- pnpm

复制 `.env.example` 为 `.env`，填入必要配置：

- `LLM_API_KEY`：LLM 服务密钥
- `LLM_MODEL`：默认模型
- `LLM_PLANNING_MODEL`：可选，用于路线蓝图和一次有界重排
- `LLM_COPY_MODEL`：可选，用于结果文案生成
- `LLM_DURATION_MODEL`：可选，用于规划前批量估计地点游玩时长
- `LLM_PLANNING_THINKING`：可选，规划模型是否开启 thinking，默认 `true`
- `LLM_COPY_THINKING`：可选，文案模型是否开启 thinking，默认 `false`
- `LLM_DURATION_THINKING`：可选，时长模型是否开启 thinking，默认 `false`
- `LLM_PLANNING_REASONING_EFFORT`：可选，规划 reasoning effort，默认 `high`
- `LLM_PLANNING_MAX_TOKENS`：可选，规划输出额度，默认 `16384`
- `LLM_COPY_MAX_TOKENS`：可选，文案输出额度，默认 `4096`
- `LLM_DURATION_MAX_TOKENS`：可选，批量时长输出额度，默认 `4096`
- `LLM_MAX_OUTPUT_TOKENS`：可选，规划 JSON 截断后可扩大的上限，默认 `32768`
- `LLM_TRUST_ENV`：可选，设为 `false` 时忽略系统代理环境
- `AMAP_API_KEY`：高德 Web 服务密钥

后端会自动读取项目根目录的 `.env`。

启动后端：

```bash
cd backend
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
uvicorn main:app --reload --port 8000
```

启动前端：

```bash
cd frontend
pnpm install
pnpm run dev
```

浏览器打开 `http://localhost:3000` 即可开始使用。

## 跨电脑或线上部署补充

当前仓库默认面向**本地前后端一起运行**。如果前端部署到另一台电脑、云主机或托管平台，需要额外补一项后端地址配置：

- 在前端运行环境中设置 `BACKEND_API_BASE_URL=https://你的后端地址`
- 不要只部署静态前端页面；`/api/backend/*` 依赖 Next.js 服务端代理，纯静态托管不会转发到 FastAPI
- 如果浏览器会直接访问 FastAPI，而不是先经过前端代理，再把后端 `CORS_ORIGINS` 改成实际前端地址

如果不配置 `BACKEND_API_BASE_URL`，前端在本地开发时会默认转发到 `http://127.0.0.1:8000`；但在其他电脑或线上环境，这个地址通常只会指向部署机器自己的本地回环地址，最终表现为“请求失败”。

## 项目结构

- `frontend/`：前端页面、组件和前端代理
- `backend/`：后端接口、地点整理、路线生成和校验
- `database/`：数据库结构参考
- `examples/`：样例输入输出
- `docs/ARCHITECTURE.md`：更完整的架构说明

## 当前边界

- 当前版本以本地运行和完整规划闭环为重点
- 输入范围聚焦用户主动提供的文本资料
- 地图能力以高德地点链接为主
